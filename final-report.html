Game Concept

To what extent did your game concept change from initial concept to what you implemented? If it did change, how did it change and why?

  • Our initial game concept was a Five Nights at Freddy’s x Night at the Museum game where users get chased by artifacts at night.
  • Eventually, the concept evolved into a ‘security guard vs thieves’ game.
  • This change likely resulted from iterative brainstorming such as “what if we added Z too” or “instead of X, why not try Y instead.”
  • Personally, I'm glad we moved away from the horror concept—I don’t mesh well with anything horror-related :)

Design

How does your final project design compare to the initial design, and what are the reasons for the differences, if any?

  • The final design was mostly faithful to our original vision.
  • We couldn't include another male thief due to time constraints.
  • The security guard was originally meant to use multiple weapons, but was simplified to a magic ability without visual effects.
  • We planned to emphasize the artifact storyline and motives for theft, but didn’t implement that part.

Schedule

How does your final schedule compare with your projected schedule, and what are the reasons for the differences, if any?

  • Our projected schedule was too optimistic.

    • Underestimating feature complexity

  • We aimed to finish by Week 9 and use Week 10 for testing, but completed the game 20 minutes before the presentation.

Development Environment

Describe your development environment. What tools did you use? What was your build workflow? If you supported multiple platforms, how did you support making your project work on all platforms? Do you have any tips or suggestions for future groups for their development environment?

  • Development environments: Visual Studio on Windows, VS Code on macOS
  • Used CMake to manage the build process
  • Maintained cross-platform support with preprocessor macros (e.g., #ifdef _WIN32)
  • Regularly tested on both OS platforms to catch platform-specific bugs
  • Suggestions for future groups:
    • Keep platform-specific code isolated and well-documented
    • Use Git for version control to prevent merge conflicts

Group Mechanics

What group mechanics decisions worked out well, and which ones (if any) did not? Why?

  • We held group meetings over Discord, but remote meetings limited early bonding.
  • Audio/technical issues occurred that would’ve been avoided with in-person meetings.
  • We structured work into modules, which caused integration problems later.

Challenges

Which aspects of the implementation were more difficult than you expected, and which were easier? Why?

  • 3D model export/import was more complex than expected, especially coordinating formats between modeling and graphics teams.
  • Assimp’s data structure for bones/animations was non-intuitive.
  • Cross-platform consistency was a constant challenge—small assumptions caused significant delays.
  • ImGUI layout was easy to use, but we had to constrain to fullscreen to avoid misalignment.
  • Integrating Miniaudio and SoLoud for sound took several hours to work properly.
  • The 2D mini-game was easier to implement in isolation, but difficult to merge due to:
    • OpenGL version mismatches (2.1 vs 3.3)
    • Upside-down textures during merging
    • Needing to adjust parameters like gravity and scaling post-integration
  • Server/client debugging was harder than anticipated due to unfamiliarity with the code and firewalls.
  • We simulated NPC behavior using randomness rather than AI—simple and effective.

Difficult Problems

What was the most difficult software problem you faced, and how did you overcome it (if you did)?

  • April: Had to integrate unfamiliar code into the server/client model. Solved it by studying and running the original code to understand it thoroughly.
  • Shuyang: OpenGL Immediate Mode was unsupported. GPT helped identify the issue. Fixed by switching to Core Profile (VAO, VBO, shaders).
  • Jessica: Transitioning from local gameplay to networking was tough. Gained understanding by reading and testing the networking code.
  • Angie: Misunderstood client code, causing bugs during server/client integration. Adjusted to send server coordinates and scale on client, though bugs remained.
  • Sue: Debugging animations in Assimp was difficult. Used various strategies (CPU validation, color-coded debugging) and found format issues (e.g., mesh as bone, unsupported Blender methods).
  • Jessie: Integration on Windows was hard due to late testing. Game size and compile time made it difficult to debug effectively under time pressure.

Toolchain

Please detail your tool chain for modeling, exporting, and loading meshes, textures, and animations. What is your opinion of the tools you used?

  • Blender used for modeling, texturing, animation. Required baking textures into images for compatibility. Preferred for being free and easier than Maya.
  • Assimp used to import models, animations, textures. Traversal logic and matrix calculations are essential for rendering hierarchy correctly.
  • Issues encountered:
    • Meshes inside meshes cause missing nodes in FBX exports.
    • Assimp structure can be unclear—must traverse nodes, not just meshes.
    • Animation debugging was difficult and tedious.
    • Recommended to use vertex weights, not meshes as bones.
    • Paths must be exported as relative; Windows paths cause Mac compatibility issues.
  • Documentation was lacking—suggest learning from past users or examples.

Libraries

Which libraries did you use and would you use them again if you were starting over? What lessons did you learn using them?

  • Networking: Used raw sockets instead of Boost due to Boost’s poor documentation. Raw sockets were educational and helped us understand networking layers better.
  • GUI: Used ImGui. Once set up, it was intuitive. Important to swap buffers only once after game and GUI rendering to avoid flickering.
  • Audio: Started with Miniaudio but switched to SoLoud (Miniaudio backend). Miniaudio only supports WAV, so we used Audacity to convert files. Found sound assets from YouTube Audio Library.

Group Dynamics

What lessons about group dynamics did you learn about working in such a large group over an extended period of time on a challenging project?

  • Splitting into smaller working groups helped with parallel development.
  • Communication was key; delays often stemmed from misunderstandings or assumptions.
  • Using Discord channels to separate topic threads (UI, graphics, netcode, testing) worked well.
  • In-person syncs could have accelerated progress during integration-heavy weeks.

Retrospective

Looking back over the past 10 weeks, is there anything you would do differently, and what would you do again in the same situation?

  • Would test the game on Windows earlier in the quarter.
  • Would finalize and merge code earlier to avoid last-minute issues.
  • Would still go with our project concept—it made the project exciting and meaningful.
  • Would again use Discord and GitHub, which were effective collaboration tools.

Advice

What advice/tips/suggestions would you give students who will take the course next year?

  • Start integration as early as possible, even if the pieces aren’t fully polished.
  • Regularly test the entire system, not just your part.
  • Use version control carefully and branch responsibly.
  • Stick to a weekly schedule and overcommunicate assumptions.
  • Playtest with users early for feedback.

Suggestions

Do you have any suggestions for improving the course? Any other comments or feedback?

  • Would be helpful to have access to B270 during the last two weekends (especially with keycard access or longer hours).
  • Offering tutorial sessions or example code for integrating major subsystems (e.g., sound, animation) could reduce the initial learning curve.
